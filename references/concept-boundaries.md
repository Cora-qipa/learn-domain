# 概念边界机制详解 / Concept Boundary Mechanism

> v2 核心:把"划清概念边界"从可选配菜,变成教学引擎的默认主菜。  
> 本文档定义边界类型、混淆风险判断、切分话术,供 `domain-mapping.md` 和 `teaching-engine.md` 调用。

---

## 一、为什么"边界"是一等公民

**初学者的真实困惑,80% 不在"A 是什么",而在"A 和 B 到底差在哪"。**

他们脑子里是一团糊在一起的概念,需要有人拿刀把它们**切开、摆正位置**。v1 把世界建模成"一堆要讲的概念",v2 把世界建模成"**概念之间的边界网络**"。

```
  初学者脑内              "划清边界"要做的
  ═══════════             ═══════════════════
   CPU GPU                   CPU │ GPU
   显卡显存      ──切──▶    ────┼────
   芯片处理器                显卡│显存│芯片
  (一团浆糊)               (每个归位、有边界)
```

## 二、概念关系四类型

每对相关概念 (A, B),其关系必属于以下之一:

### ① 并列 (Parallel)
**定义**: A 和 B 是同级的两个东西,各干各的,不是谁包含谁。

**识别标志**:
- 功能/目标不同,但常被拿来比较
- "A 适合做 X,B 适合做 Y"
- 经常出现"A vs B"的对比文章

**例子**:
- CPU ∥ GPU (同级处理器,一个少而精、一个多而傻)
- 股票 ∥ 债券 (同级金融工具,风险收益不同)
- GAN ∥ 扩散模型 (同级生成模型,技术路线不同)

**切分话术模板**:
> "A 和 B 的本质区别:A 是 ___(特点)___,B 是 ___(特点)___。  
> A 适合 ___,B 适合 ___。"

**易错点**:新手常误以为能互换,或不知道各自适用场景。

---

### ② 包含 (Containment)
**定义**: A 是 B 的一部分,B 由 A + 其他东西组成。

**识别标志**:
- 物理上:A 装在 B 里
- 逻辑上:B = A + X + Y
- "B 包含 A"或"A 是 B 的核心"

**例子**:
- GPU ⊂ 显卡 (GPU 是芯片,显卡是 GPU + 显存 + 供电 + 散热的整块板)
- 发动机 ⊂ 汽车
- React ⊂ 前端框架生态

**切分话术模板**:
> "A 是 B 的一部分。B = A + ___。  
> 打个比方:A 之于 B = ___ 之于 ___。"  
> (类比要标边界:"简化之处:___")

**易错点**:新手最容易把"包含"误当"并列",以为是两个独立的东西。

---

### ③ 层级 (Hierarchy / Dependency)
**定义**: A 是 B 的底层/上层,存在依赖关系(谁指挥谁、谁建立在谁之上)。

**识别标志**:
- "A 控制/驱动/指挥 B"
- "B 基于/依赖 A"
- 有时间/流程上的先后

**例子**:
- CUDA → GPU (软件指挥硬件)
- CPU → GPU (总指挥 → 计算工厂)
- 训练 → 推理 (先学后用,有时间先后)
- 操作系统 → 应用程序

**切分话术模板**:
> "A 和 B 不是并列,而是 ___ 关系:  
> A ___(动词:指挥/驱动/建立在)___ B。  
> 没有 A,B ___。"

**易错点**:新手不知道谁依赖谁,或误以为是并列。

---

### ④ 易混淆 (Confusable / Catch-all)
**定义**: A 和 B 关系不明显属于①②③,但在**实际使用中新手极易混淆**的兜底分类。

**识别标志**:
- 名字相似 / 经常共现 / 口语混用
- 不同场景下指代不同但名字相同
- 关系复杂,不好用一刀切

**例子**:
- 训练 ⟷ 推理 (都是"跑模型",但一个是学、一个是用)
- 显存 ⟷ 内存 (都是"存东西",但一个给 GPU 用、一个给 CPU 用)
- "模型"在不同场景 (机器学习的模型 vs 3D 建模 vs 商业模式)

**切分话术模板**:
> "A 和 B 听起来像 / 经常一起出现,但其实:  
> A 是 ___,B 是 ___。  
> 关键区别:___。"

**注意**: ④ 是**兜底**,优先尝试归到①②③。只有真正"关系不好归类但确实常混"的才放这。

## 三、混淆风险判断(启发式)

每对概念除了标关系类型,还要判断**混淆风险**(高/中/低)。

### 混淆风险 = 高
满足以下任一条:
- ✅ 名字相似(如 CPU/GPU,都带 PU)
- ✅ 功能部分重叠 + 新手常问"啥区别"
- ✅ 口语中经常混用(如"显卡"和"GPU")
- ✅ 包含关系但新手常误当并列

**处理**: 阶段1 标 ⚠️,阶段3 **强制切、优先切、切得更狠**。

### 混淆风险 = 中
- 关系清晰,但可能被误解
- 不常共现,但问起来会卡

**处理**: 阶段3 讲到时轻提边界。

### 混淆风险 = 低
- 关系明显,不太会混

**处理**: 可选,用户主动问才展开。

### 判断流程
```
概念对 (A, B)
  │
  ├─ 1. 先判关系类型(①②③④)
  │
  └─ 2. 再判混淆风险
       ├─ 名字像 / 功能重叠 / 口语混用? → 高
       ├─ 关系不够直观? → 中
       └─ 否则 → 低
```

**迭代优化**: 
- v2.0 用上述启发式自动判断
- 遇到误判(该切没切、不该切强切了),人工校正
- 校正后沉淀到混淆风险库(v2.1 再做)

## 四、边界切分话术库

### 通用结构
```
边界切分 = 一句话本质 + (可选)类比 + 各自适用场景

① 一句话本质 (必须)
   "A 和 B 的根本区别:A 是 ___, B 是 ___。"
   
② 类比 (可选,能降低理解门槛时用)
   "打个比方:A 之于 B = X 之于 Y。"
   ⚠️ 类比必须标边界:"简化之处:___"
   
③ 各自适用场景 (必须)
   "A 适合 ___,B 适合 ___。"
   或"有 A 才能 ___,有 B 才能 ___。"
```

### 按关系类型的话术
见上面各类型的"切分话术模板"。

### 反面教材:不要这样切
❌ **参数罗列无本质**
> "CPU 核心少、单核强;GPU 核心多、单核弱。CPU 适合串行,GPU 适合并行。"
(列了 4 个维度,但没有"一句话抓住本质")

✅ **应该这样切**
> "CPU 和 GPU 的根本区别:**少而精 vs 多而傻**。CPU 几个超强核心处理复杂任务,GPU 几千个简单核心处理海量重复计算。"
(一句话抓住本质,其他维度自然推导)

## 五、在教学流程中的应用

### 阶段1 · 测绘地形时
1. 拆解领域时,不只列概念,还标**概念对的关系**(①②③④)
2. 识别**高混淆风险对**,在地图上标 ⚠️
3. 用简单 ASCII 图可视化关系网(见 `domain-mapping.md`)

### 阶段3 · 讲解时
遇到高混淆对 (A, B):
```
① 先切边界 (强制,前置到各自展开之前)
   用"一句话本质 + 适用场景"
   
② 再各自展开
   A: 点 → 线-纵 → (线-横已在①做过,不重复)
   B: 点 → 线-纵
   
③ 关卡确认 (验证边界是否真清楚)
   反问:"你能说出它俩最关键的不同吗?"
```

**关键**: "线-横"不再是可选配菜,而是**必经环节且前置**。

## 六、示例:CPU vs GPU

### 关系标注
- **类型**: ① 并列
- **混淆风险**: 高(名字相似、功能部分重叠、新手常问)

### 边界切分(阶段3 强制讲)
```
一句话本质:
  "CPU 和 GPU 的根本区别:少而精 vs 多而傻。"

展开:
  CPU = 几个超级聪明的博士,核心少但每个极强,
        擅长处理复杂、环环相扣的任务。
  GPU = 几千个只会算术的小学生,每个都弱,
        但人海战术同时算,擅长海量简单重复计算。

类比(标边界):
  "打个比方:1000 道简单加减法,
   博士(CPU)得一道道排队做,
   小学生(GPU)每人分 1 道,一瞬间全做完。
   ⚠️ 简化:反过来,1 道极烧脑的题,
   1000 个小学生反而抓瞎,只有博士能干。"

各自适用:
  游戏画面、AI 矩阵运算 = 海量简单计算 → GPU
  操作系统调度、复杂逻辑 = 环环相扣任务 → CPU
```

### 关卡确认
反问:"你能说出 CPU 和 GPU 最关键的不同吗?"
(期望听到"少而精 vs 多而傻"或"复杂任务 vs 简单大量")

## 七、常见概念对速查表

供测绘阶段参考(领域不同,这张表会变):

| 概念对 | 关系 | 混淆风险 | 一刀本质 |
|---|---|---|---|
| CPU ⟷ GPU | ① 并列 | 高 | 少而精 vs 多而傻 |
| GPU ⟷ 显卡 | ② 包含 | 高 | 芯 vs 整块板 |
| 显存 ⟷ 内存 | ① 并列 | 中 | 给GPU的料 vs 给CPU的料 |
| CUDA ⟷ GPU | ③ 层级 | 中 | 软件指挥 vs 硬件干活 |
| 训练 ⟷ 推理 | ③ 层级 / ④ 易混 | 高 | 学 vs 用 |
| 芯片 ⟷ 处理器 | ④ 易混 | 低 | 物理实体 vs 功能称呼 |

(此表随实践迭代,v2.1 可沉淀成独立库)

## 八、反馈与迭代

### 用户反馈收集点
讲完一个领域后,可选问:"刚才有哪对概念,你觉得还是有点混?"
收集到的 → 标记为高混淆,下次优先切。

### 误判类型
- **假阳性**:不该强制切的被切了,用户觉得啰嗦 → 降低混淆风险
- **假阴性**:该切的没切,用户后来卡住 → 提升混淆风险

校正后沉淀,逐步建立领域混淆风险库。
