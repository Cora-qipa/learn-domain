# learn-domain v2 提案:概念边界优先 / Concept Boundary-First

> **状态**: 提案阶段 · Proposal Phase  
> **提出时间**: 2026-06-01  
> **基于**: v1.0.0 真实用户测试反馈(英伟达/芯片领域走查)

---

## 一、问题陈述 / Problem Statement

### v1 暴露的核心缺陷

**真实用户反馈**:"不够本质。比如不会主动跟我理清楚 CPU 和 GPU 的本质区别。概念间的本质区别对初学者而言是重要的。"

**根本原因**:v1 把世界建模成"一堆要讲的概念"(逐个讲清),而初学者的真实困惑是"**概念和概念之间的那条线在哪**"。初学者脑子里是一团糊的浆糊,需要有人拿刀把它们**切开、划清边界**,而不只是逐个定义。

```
  v1 的世界观                实际初学者脑内
  ══════════                ════════════════
  概念A → 概念B → 概念C      CPU GPU 显卡 显存 芯片
  (线性讲解队列)             (一团浆糊,边界不清)

                            真正需要的
                            ═══════════
                               CPU │ GPU
                              ─────┼─────
                              显卡 │ 显存 │ 芯片
                            (切开、归位、有边界)
```

**当前设计漏洞**:
1. "线-横"(概念对比)在 v1 里是**可选配菜**,只在模式①轻量带,不是必经环节。
2. 对比时机是**被动响应**(用户问了才讲),而不是主动识别"这两个新手必混"并强制切。
3. 没有"边界"这个一等公民——只有"概念"这个实体,概念间的**关系**没有被显式建模。

---

## 二、核心洞察 / Core Insight

### 概念关系三类型(客观) + 混淆风险(主观)

初学者的边界困惑,本质来自**三种概念关系**没理清:

| 关系类型 | 含义 | 例子 | 混淆表现 |
|---|---|---|---|
| **① 并列** | A 和 B 同级,各干各的 | CPU ∥ GPU | 误以为能互换 / 不知道各自适合什么 |
| **② 包含** | A 是 B 的一部分 | GPU ⊂ 显卡 | 误以为是两个东西 / 不知道谁装谁 |
| **③ 层级** | A 是 B 的底层/依赖 | CUDA → GPU | 不知道谁指挥谁 / 缺了谁不行 |

**关键修正**(探索阶段的发现):
- 原本提出的"④ 易混边界"**不是第四种类型**,而是①②③在新手那里"没画清"导致的症状。
- "易混"是**主观学习障碍**,不是客观关系。任何一种关系(并列/包含/层级)都可能"高混淆风险"。

**v2 应建模的**:
```
概念对 (A, B)
  ├─ 关系类型: 并列 / 包含 / 层级  (客观,领域本身决定)
  └─ 混淆风险: 高 / 中 / 低        (主观,新手经验/数据)
      └─ 高风险 → 强制切、优先切、切得更狠
```

## 三、v2 目标 / Goals

### 主目标
**让"划清概念边界"从可选配菜,变成教学引擎的默认主菜。**

### 具体目标
1. ✅ **强制辨析**:每对高混淆风险的概念,必须主动切清边界(不等用户问)。
2. ✅ **边界优先于定义**:先给"A 和 B 的本质区别是 ___",再展开各自细节。
3. ✅ **可视化边界网**:阶段1 给用户一张"概念关系图",显式标出并列/包含/层级关系。
4. ⚠️ **混淆风险库**:沉淀"哪些概念对容易混"的经验库(可选,v2.1 再做)。

### 非目标(明确不做)
- ❌ 跨会话记忆(留 v3)
- ❌ 插件市场订阅化(已在 roadmap,但与边界机制无关)
- ❌ 每个概念讲得更深(那是纵向挖,我们聚焦横向切)

## 四、设计方案 / Design

### 4.1 阶段1 增强:测绘"概念边界网"

**现状**(v1):阶段1 输出依赖图 + 80/20 最小集,纯文字描述。

**v2 改造**:
```
阶段1 产出:概念边界网
  ┌────────────────────────────────────────┐
  │  [领域] 概念关系图                      │
  ├────────────────────────────────────────┤
  │                                        │
  │   CPU ═══(并列)═══ GPU                 │
  │    │                │                  │
  │  (层级)           (包含)               │
  │    ↓                ↓                  │
  │  内存             显卡 ⊃ [GPU + 显存]   │
  │                                        │
  │  ⚠️ 高混淆: CPU↔GPU, 显卡↔GPU          │
  └────────────────────────────────────────┘
```

**新增动作**:
1. 拆解领域时,不只列概念,还标**概念对的关系**(并列/包含/层级)。
2. 识别**高混淆风险对**(基于启发式:名字相似/经常共现/口语混用)。
3. 用简单 ASCII 图可视化关系网,让用户先看到"边界地图"。

### 4.2 阶段3 改造:边界优先讲解

**现状**(v1):逐个概念,走 点 → 线-纵 → 线-横(可选)。

**v2 改造**:
```
讲解单元从"单个概念"变成"概念对/簇"

遇到高混淆对 (A, B):
  ① 先切边界(强制)
     "A 和 B 的本质区别:___ vs ___"
     一句话抓住那一刀,不是列5个维度让用户自己悟
  
  ② 再各自展开
     A: 点 → 线-纵 → (线-横已在①做过)
     B: 点 → 线-纵
  
  ③ 关卡确认
     反问:"你能说出它俩最关键的不同吗?"(验证边界是否真清楚)
```

**关键变化**:
- "线-横"从配菜提升为**必经环节**,且**前置**到两个概念各自展开之前。
- 对比的重心从"参数表格"变成"**一句话本质区别**"。

## 五、实施计划 / Implementation Plan

### 分阶段交付

| 阶段 | 任务 | 产出 | 验收标准 |
|---|---|---|---|
| **P1** 理论落地 | 写 `concept-boundaries.md` + 改 `domain-mapping.md` | 边界三类型 + 混淆识别 | 文档自洽、可操作 |
| **P2** 引擎改造 | 改 `teaching-engine.md` + 修 SKILL.md | "概念对优先"讲解单元 | 跑通一个领域(如芯片),对比 v1 |
| **P3** 可视化 | ASCII 关系图生成逻辑 | 阶段1 输出边界网 | 图能看懂、关系清晰 |
| **P4** 示例 + 测试 | 新增 example + 回归测试 | v2 example + 触发矩阵不退化 | 真实用户走查 ≥ v1 |
| **发布** | 打 v2.0.0 tag + 更新 README/CHANGELOG | GitHub 发布 | 陌生人 clone 可用 |

### 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| "边界网"绘制复杂度高,慢 | 违背"快速入门" | P3 先做简化版 ASCII,复杂可视化留 v2.1 |
| "混淆风险"识别不准,误判 | 强制切了不该切的,啰嗦 | 启发式保守(只标名字/功能极相似的),可选关闭 |
| 改动较大,v1 用户不适应 | 回归风险 | P4 必须真实用户走查,对比 v1 体感 |

## 六、成功标准 / Success Criteria

### 定性
真实用户(非作者)走完 v2 后,**主动说出类似**:
> "这次每个概念都很清楚它和别的概念区别在哪,没有之前那种'好像懂又好像混'的感觉。"

### 定量
用 scenario-rubric.md 评分,v2 在同一领域应满足:
- "先诊断后讲"等既有 7 项不退化
- **新增第 8 项**:"高混淆概念对,是否主动切清边界"(1 分)
- 3 领域均值 ≥ 7/8

## 七、后续演进 / Future Work

v2 聚焦"边界",以下留给更远期:

- **v2.1 混淆风险库**:沉淀"芯片领域 CPU↔GPU 高混、金融领域 股票↔债券 高混"这种领域知识,变成可复用资产。
- **v3 跨会话记忆**:记住用户已学的概念和边界,下次直接从断点继续(最大的范式跳跃)。
- **可视化增强**:从 ASCII 升级到真实的交互式关系图(需前端组件,工程量大)。

---

## 八、决策点 / Decision Points

需要用户拍板:

1. ✅ **边界三类型(并列/包含/层级)是否够用?** 还是要再补其他类型?
2. ⚠️ **"混淆风险"由谁判断?** 
   - 方案 A:skill 内置启发式(名字相似/共现/口语混用) — 快但可能误判
   - 方案 B:领域专家标注 → 人工重、不可扩展,但准
   - 方案 C:先 A,遇到误判再人工校正沉淀到库
3. ⚠️ **边界网可视化做到什么程度?**
   - 最小:纯文字"A 和 B 是并列、C 包含 D"
   - 适中:简单 ASCII 图(P3 目标)
   - 理想:交互式图(v2.1+)

### v1 讲"CPU vs GPU"(被动,用户问了才讲)
```
[讲 CPU] → [讲 GPU] → (用户问"它俩啥区别") → 给表格对比
```

### v2 讲"CPU vs GPU"(主动,识别高混淆)
```
[阶段1 标出"CPU ∥ GPU,高混淆"]
  ↓
[阶段3 遇到这对,先切边界]
  "CPU 和 GPU 本质区别:少而精 vs 多而傻"
```
